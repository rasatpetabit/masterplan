## 2026-06-06 — Guard D: NFS-safe cross-session owner sentinel (Phase 3b) — provisional-acquire, live-perfect / stale-benign — SHIPPED in v8.1.0 (7e604a5, pushed)

Built the never-shipped Guard D from the worktree/concurrency hardening plan (task #6): mutual exclusion for two LLM sessions operating the **same run bundle** concurrently (epyc1/epyc2, NFS). `state.yml` can't provide it (`writeState` is an atomic write, not test-and-set), so this is a dedicated NFS-safe lock.

**Identity = SESSION, not process.** Every `mp` call is an ephemeral OS process (pid/start-time churn each call; `kill(pid,0)` always dead), so the stable anchor is `{host, CLAUDE_CODE_SESSION_ID}` (a UUID → no PID-reuse guard needed). Liveness is heartbeat-age TTL only (default 30m; must exceed the max single background wave, since the LLM only heartbeats at turn/wave boundaries). **Two files (double-ownership guard):** immutable `.owner.lock` (atomic `link()`-create, confirmed via `stat().nlink===2` — NOT link()'s return, which some NFS clients mis-report after a transparent retry) + per-owner `.owner.hb.<host>.<session>` the owner refreshes for ITSELF only (never rewriting the lock → a stalled owner's late heartbeat can't clobber a successor).

**The honest guarantee (NOT "perfect single-writer" — that's impossible on NFS without a lock manager):** Guard D gives **PERFECT mutual exclusion for LIVE contention** — a fresh contended lock is an atomic `link()` create, so two live sessions never both proceed (the loser cleanly resolves to `blocked`). The **unit of protection is the TURN**, not the individual write: the shell re-heartbeats at every turn entry (step 1.6) and before the completion write (§2a step 0), and under live contention exactly one contender resolves to `held-by-self` while the rest get `lost-to-other` and STOP. The **one residual, accepted by design:** a `>TTL`-abandoned owner (silent 30+ min) that RESURRECTS at the exact instant a reclaimer breaks its lock can double-write. Narrow, benign, documented — not a gap to close with another mechanism.

**The hard part — the stale-break storm, and why acquire is PROVISIONAL (design, dead-ends, resolution):** a stale lock must be removed before the empty path can be re-`link()`'d, but there is **no atomic "remove path P iff it still resolves to inode I" on NFS** (or POSIX). So the removal is unavoidably path-based (`renameSync(lockPath, .dead.*)`), and under a concurrent break storm a late breaker can evict a FRESH successor and let a 2nd acquirer slip onto the now-empty path — i.e. **two sessions can transiently report `acquire`/`steal`**. Reproduced: N=60 yielded 2 reported winners ~1-in-5 (the on-disk lock stayed single, so the safety property already held — but the *reported* outcome over-promised). **Dead-ends explored + rejected (4 iterations):** (1) a generation-named break-token won by exclusive `link()` — Codex (gpt-5.5, xhigh) found 3 BLOCKERs (release+reacquire still evicts a fresh successor; `forceBreak` unserialized; crash-leaked token wedges the generation). (2) a mutation-mutex M — its leak self-heal needs a small TTL that breaks cross-host clock-skew robustness. advisor() confirmed each is a subtler version of the same NFS impossibility. **Resolution — stop trying to make acquire exactly-once; make it honestly PROVISIONAL.** Dropped the break-token; `stealBreak` is a plain generation+TTL-gated rename-away whose generation check only **reduces churn**, never claims safety. The heartbeat re-check (`heartbeatOwner` → `held-by-self` for *exactly* the on-disk owner) is the boundary that **resolves transient multi-winner churn into a single proceeding writer** — the shell runs it after a provisional win (step 1.6 confirm branch) and before every state write (§2a step 0). **Evidence:** stress invariant relaxed to the correct one — acquire-winners ≥1 (churn allowed), but **`held-by-self==1` + `lost-to-other==N-1`** at the heartbeat boundary; N=60 yields `hbs==1`.

**Codex round-2 review (provisional-acquire focus, brief `/tmp/codex-guard-d-review2.txt`) — 2 BLOCKER + 1 MAJOR, resolved:** (a) **BLOCKER-1 (real): `heartbeatOwner` read→write TOCTOU** — a steal landing between the initial read/decide and the state write let a displaced owner write off a stale `held-by-self`. **Fix 1 (applied):** `heartbeatOwner` now WRITE-CONFIRMS — refresh our hb first (makes a concurrent stale-break re-confirm see us fresh and bail), then RE-READ `.owner.lock` and claim `held-by-self` only if it's STILL ours; else unlink the hb we just wrote and return `lost-to-other`. advisor() confirmed this shrinks the window to the same benign stale-takeover residual. (b) **BLOCKER-2 (claim-defect, not a machinery gap): "the shell doesn't heartbeat before every write"** — advisor ruled the fix is to document the **turn-scoped** guarantee, NOT to build a `writeState` ownership chokepoint (iteration-5 trap: an env-reading check on the universal write path to chase a benign residual). Verified turn-entry coverage: step 1.6 fires on every §2 entry, §2a step 0 on completion. (c) **MAJOR: `releaseOwner` unlink race** — read-at-top then unlink-later could clobber a successor that stole+recreated the lock in the window. **Fix 3 (applied):** re-read immediately before the unlink; only remove a lock still `isSelf`. **Both fixes proved by deterministic fs-wrapper unit tests** (inject a steal between the read and the post-write/pre-unlink re-read; the `nlink===2` test is the template). The "authoritative single-writer gate" language was the over-claim Codex's BLOCKER-2 keyed on — **reframed everywhere** to live-perfect / stale-benign (owner.mjs + owner-fs.mjs headers, masterplan.md step 1.6 + §2e¶8, owner-stress.sh, this entry).

**Codex round-3 review (the two surgical Fix-1/Fix-3 fixes, brief `/tmp/codex-guard-d-review3.txt`) — 1 BLOCKER (real), resolved; Fix-1 CLEARED:** Codex cleared Fix-1 (`heartbeatOwner` write-confirm closes the read→write TOCTOU; residual is strictly the documented stale-takeover case). But it found Fix-3's `releaseOwner` **re-read-then-path-unlink still races**: the re-read only protects the window BEFORE it — the path-`unlink(.owner.lock)` after `isSelf(before)` still fires, so a successor B that steals our STALE lock and recreates it in the window AFTER our re-read but BEFORE our unlink gets its FRESH lock clobbered → strands a live successor + opens a B+C double-write. **Root-cause (the consistency insight, advisor-confirmed):** `acquire`, `heartbeat`, and `stealBreak` all consult the staleness predicate — **only `releaseOwner` blindly path-unlinked**, the one lock-op that bypassed it. **Fix (freshness gate, applied):** `releaseOwner` now path-unlinks ONLY a lock proven still WITHIN TTL — a successor can only steal a STALE lock, so fresh ⟹ no successor can exist to clobber; a stale owned lock returns the new `stale-not-released` outcome and is **left for the reclaim path** (a stale lock is exactly what a reclaimer is entitled to break). ttlEff mirrors `acquireOwner` (ttlMs when threaded, else `OWNER_TTL_DEFAULT_MS`) so release + acquire share ONE staleness predicate; the gate engages whenever `now` is known (bin defaults `--now` to `Date.now()` → engaged in every production release). **advisor's implementation catch:** the freshness sample reads our own hb via `readLastHeartbeat`, but the old code deleted that hb EARLY → a post-delete sample falls back to `acquiredAt` → false "stale" → A refuses to release its OWN fresh lock → strands the bundle. **Resolution:** sample freshness BEFORE deleting the hb; keep the hb on disk through the critical window (load-bearing — a concurrent breaker re-reads it and bails when we're fresh); on the release path unlink the LOCK first, our hb LAST.
